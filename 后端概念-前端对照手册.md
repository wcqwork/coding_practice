## 1. 整体架构对照

### 前端项目典型结构

```
src/
├── api/                  ← 发请求的函数（axios 封装）
├── store/                ← 业务逻辑（Pinia / Vuex）
├── pages/                ← 页面组件
├── components/           ← UI 组件
└── types/                ← TypeScript 类型定义
```

### 后端项目结构（本项目）

```
project/src/main/java/com/kakarote/project/
├── controller/           ← ≈ pages/（接收请求、返回响应）
│   └── ProjectsController.java
├── service/              ← ≈ store/（业务逻辑）
│   ├── IProjectBulletinService.java        ← 接口（定义有哪些方法）
│   └── impl/
│       └── ProjectBulletinServiceImpl.java ← 实现（具体逻辑）
├── mapper/               ← ≈ api/（和数据库打交道）
│   ├── ProjectBulletinMapper.java          ← Java 接口（定义方法签名）
│   └── xml/ProjectBulletinMapper.xml       ← XML（写具体的 SQL）
```

### 请求流转全链路

```
前端发起请求  POST /projects/saveBulletinLeader
    │
    ▼
┌─────────────────────────────────────────────────┐
│  Controller（相当于页面的事件处理函数）            │
│  1. 接收参数                                      │
│  2. 校验参数（if (projectId == null) 报错）       │
│  3. 调用 Service                                  │
│  4. 把结果包装成统一格式返回                       │
└─────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────┐
│  Service（相当于 Pinia Store 的 action）          │
│  1. 处理业务逻辑（截断文本、判断条件等）           │
│  2. 调用 Mapper 操作数据库                        │
│  3. 返回处理结果                                   │
└─────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────┐
│  Mapper（相当于 api/ 下的请求函数，但目标是数据库）│
│  1. Java 接口定义方法签名                         │
│  2. XML 文件编写具体 SQL                          │
│  3. 框架自动将 SQL 结果映射为 Java 对象            │
└─────────────────────────────────────────────────┘
    │
    ▼
  数据库（MySQL）
```

## 2. 注解（Annotation）↔ 前端类比

Java 注解是 `@` 开头的标记，类似于前端的 **装饰器（Decorator）** 或 **组合式函数标记**。

| Java 注解 | 前端类比 | 作用 |
|-----------|---------|------|
| `@PostMapping("/xxx")` | `app.post('/xxx', handler)` | 注册一个 POST 路由 |
| `@Autowired` | `const store = useXxxStore()` | 自动获取一个已注册的服务实例 |
| `@Service` | `export const useXxxStore = defineStore(...)` | 把这个类注册为全局可注入的服务 |
| `@Data` (Lombok) | 自动生成 getter/setter | 省去手写 `getName()` / `setName()` |
| `@TableName("xxx")` | Prisma 的 `@@map("xxx")` | 指定这个类对应数据库的哪张表 |
| `@TableId(type = IdType.AUTO)` | Prisma 的 `@id @default(autoincrement())` | 标记主键，自增 |
| `@RequestParam` | Express 的 `req.query.xxx` | 从 URL query 参数取值 |
| `@Transactional` | 无直接类比，见[第 9 节](#9-事务-transactional) | 数据库事务，保证原子性 |
| `@Async` | `setTimeout(() => {}, 0)` 或 Web Worker | 异步执行，不阻塞主线程 |
| `@Override` | TypeScript 的 `implements` 检查 | 标记这是接口方法的实现 |

## 3. Controller 层详解

### 前端类比：页面组件的事件处理 + API 路由

```java
// ==================== 后端代码 ====================

@RestController                         // ≈ 定义一个路由模块
public class ProjectsController {

    @Autowired                          // ≈ const bulletinService = useBulletinStore()
    private IProjectBulletinService projectBulletinService;

    @PostMapping("/getBulletinInfo")    // ≈ app.post('/getBulletinInfo', handler)
    public Result<Map<String, Object>> getBulletinInfo(@RequestParam Long projectId) {
        // ① 参数校验 — 类比前端表单校验
        if (ObjectUtil.isEmpty(projectId)) {
            return R.error(SystemCodeEnum.SYSTEM_NO_VALID, "项目ID不能为空");
        }
        // ② 调用 Service — 类比调用 store 的 action
        String totalLeader = projectBulletinService.getBulletinLeader(projectId);
        // ③ 组装返回数据 — 类比 return { data: { totalLeader } }
        Map<String, Object> data = new HashMap<>();
        data.put("totalLeader", totalLeader);
        return R.ok(data);
    }
}
```

```typescript
// ==================== 前端等价代码（伪代码）====================

// 用 Express 来类比
app.post('/getBulletinInfo', async (req, res) => {
    const { projectId } = req.query

    // ① 参数校验
    if (!projectId) {
        return res.json({ ret: -1, msg: '项目ID不能为空' })
    }

    // ② 调用 Service（业务逻辑）
    const totalLeader = await bulletinService.getBulletinLeader(projectId)

    // ③ 组装返回数据
    return res.json({ ret: 0, data: { totalLeader } })
})
```

### 关键对照

| 后端写法 | 前端写法 | 说明 |
|---------|---------|------|
| `@RequestParam Long projectId` | `req.query.projectId` | 从 URL 问号参数取值 |
| `R.ok(data)` | `res.json({ ret: 0, data })` | 成功响应 |
| `R.error(code, msg)` | `res.json({ ret: code, msg })` | 失败响应 |
| `Result<Map<String, Object>>` | `{ ret: number; data: Record<string, any>; msg: string }` | 返回值类型 |

### 本次提交暴露的 4 个接口

```
POST /getBulletinInfo        → 获取公告（总负责人）
POST /saveBulletinLeader     → 保存总负责人
POST /getBulletinReadStatus  → 获取未读状态（红点）
POST /markBulletinRead       → 标记已读
```

## 4. Service 层详解

### 前端类比：Pinia Store 的 Action

```java
// ==================== 后端代码 ====================

// ① 接口（Interface）—— 定义"有哪些方法"
// 前端类比：TypeScript 的 interface，或者 store 的类型定义
public interface IProjectBulletinService extends BaseService<ProjectBulletin> {
    void bumpBulletinVersion(Long projectId);
    String getBulletinLeader(Long projectId);
    void saveBulletinLeader(Long projectId, String totalLeader);
    boolean getReadStatus(Long projectId);
    void markRead(Long projectId);
}

// ② 实现类（ServiceImpl）—— 具体业务逻辑
// 前端类比：Pinia Store 的 defineStore 内容
@Service   // 告诉框架"这是一个服务，可以被注入"
public class ProjectBulletinServiceImpl
        extends BaseServiceImpl<ProjectBulletinMapper, ProjectBulletin>
        implements IProjectBulletinService {

    @Autowired
    private ProjectBulletinMapper projectBulletinMapper;

    @Override   // 表示这是接口中定义的方法的具体实现
    public void saveBulletinLeader(Long projectId, String totalLeader) {
        // 业务逻辑：截断超长文本
        String leader = totalLeader != null ? totalLeader : "";
        if (leader.length() > 100) {
            leader = leader.substring(0, 100);
        }
        // 调用 Mapper 操作数据库
        projectBulletinMapper.upsertLeader(projectId, leader);
    }

    @Override
    public boolean getReadStatus(Long projectId) {
        Long userId = UserUtil.getUserId();  // ≈ 从 token 获取当前登录用户 ID
        Boolean hasUpdate = projectBulletinMapper.getReadStatus(projectId, userId);
        return Boolean.TRUE.equals(hasUpdate);
    }
}
```

```typescript
// ==================== 前端等价代码（伪代码）====================

// ① 接口定义
interface IBulletinService {
    bumpBulletinVersion(projectId: number): Promise<void>
    getBulletinLeader(projectId: number): Promise<string>
    saveBulletinLeader(projectId: number, totalLeader: string): Promise<void>
    getReadStatus(projectId: number): Promise<boolean>
    markRead(projectId: number): Promise<void>
}

// ② 具体实现（Pinia Store）
export const useBulletinStore = defineStore('bulletin', () => {
    const api = useBulletinApi()  // ≈ Mapper

    async function saveBulletinLeader(projectId: number, totalLeader: string) {
        // 业务逻辑：截断超长文本
        let leader = totalLeader ?? ''
        if (leader.length > 100) {
            leader = leader.substring(0, 100)
        }
        // 调用 API
        await api.upsertLeader(projectId, leader)
    }

    async function getReadStatus(projectId: number) {
        const userId = getCurrentUserId()  // ≈ UserUtil.getUserId()
        const hasUpdate = await api.getReadStatus(projectId, userId)
        return hasUpdate ?? false
    }

    return { saveBulletinLeader, getReadStatus, ... }
})
```

### 关键对照

| 后端写法 | 前端写法 | 说明 |
|---------|---------|------|
| `interface + impl` 分离 | TypeScript interface + 实现 | 接口与实现分离，方便替换和测试 |
| `UserUtil.getUserId()` | `localStorage.getItem('userId')` 或 JWT 解析 | 获取当前登录用户 |
| `lambdaQuery().eq(...).one()` | `data.find(item => item.projectId === projectId)` | 链式查询（MyBatis-Plus 提供的便捷 API） |
| `projectBulletinMapper.upsertLeader(...)` | `await api.upsertLeader(...)` | 调用数据层 |

### lambdaQuery 详解

MyBatis-Plus 提供的链式查询 API，类似前端的 `Array.filter()`：

```java
// 后端
ProjectBulletin bulletin = lambdaQuery()
    .eq(ProjectBulletin::getProjectId, projectId)   // WHERE project_id = ?
    .eq(ProjectBulletin::getUserId, 0L)             // AND user_id = 0
    .one();                                          // LIMIT 1，取第一条
```

```typescript
// 前端等价
const bulletin = bulletins
    .filter(b => b.projectId === projectId)
    .filter(b => b.userId === 0)[0]  // 取第一条
```

## 5. Mapper 层详解

### 前端类比：api/ 目录下的请求函数，但目标是数据库而不是 HTTP

Mapper 分为两部分：

```
ProjectBulletinMapper.java  ← 接口，定义有哪些数据库操作方法
ProjectBulletinMapper.xml   ← XML，编写具体的 SQL 语句
```

#### 5.1 Java 接口

```java
// ==================== 后端代码 ====================

public interface ProjectBulletinMapper extends BaseMapper<ProjectBulletin> {
    // 继承 BaseMapper 后，自动拥有：selectById、insert、update、delete 等基础方法
    // 下面是自定义的数据库操作：

    void bumpVersion(@Param("projectId") Long projectId);
    void upsertLeader(@Param("projectId") Long projectId, @Param("totalLeader") String totalLeader);
    Boolean getReadStatus(@Param("projectId") Long projectId, @Param("userId") Long userId);
    void markRead(@Param("projectId") Long projectId, @Param("userId") Long userId);
}
```

```typescript
// ==================== 前端等价代码（伪代码）====================

// BaseMapper 提供的基础方法 ≈
interface BaseMapper<T> {
    selectById(id: number): Promise<T>
    insert(entity: T): Promise<void>
    updateById(entity: T): Promise<void>
    deleteById(id: number): Promise<void>
    // ...
}

// 自定义方法
interface BulletinMapper extends BaseMapper<ProjectBulletin> {
    bumpVersion(projectId: number): Promise<void>
    upsertLeader(projectId: number, totalLeader: string): Promise<void>
    getReadStatus(projectId: number, userId: number): Promise<boolean>
    markRead(projectId: number, userId: number): Promise<void>
}
```

#### 5.2 XML 编写 SQL

```xml
<!-- ==================== 后端代码 ==================== -->

<!-- 保存总负责人 + 版本号递增 -->
<insert id="upsertLeader">
    INSERT INTO pre_project_bulletin (project_id, user_id, total_leader, content_version, created_at, updated_at)
    VALUES (#{projectId}, 0, #{totalLeader}, 1, NOW(), NOW())
    ON DUPLICATE KEY UPDATE
        total_leader = #{totalLeader},
        content_version = content_version + 1,
        updated_at = NOW()
</insert>
```

逐行解释：

| SQL 语句 | 说明 | 前端类比 |
|----------|------|---------|
| `INSERT INTO ... VALUES (...)` | 插入一条新记录 | `array.push(newItem)` |
| `ON DUPLICATE KEY UPDATE ...` | 如果已存在则更新 | `map.set(key, value)` 或 `array.upsert()` |
| `#{projectId}` | 参数占位符（防 SQL 注入） | 模板字符串 `${projectId}`，但安全 |
| `content_version = content_version + 1` | 版本号自增 | `item.version++` |
| `NOW()` | 当前时间 | `new Date()` |

#### 5.3 本次提交中的 4 条 SQL

**① bumpVersion — 纯递增版本号**

```sql
INSERT INTO pre_project_bulletin (project_id, user_id, content_version, ...)
VALUES (#{projectId}, 0, 1, ...)
ON DUPLICATE KEY UPDATE content_version = content_version + 1
```

类比前端：
```typescript
function bumpVersion(projectId: number) {
    const existing = records.find(r => r.projectId === projectId && r.userId === 0)
    if (existing) {
        existing.contentVersion++
    } else {
        records.push({ projectId, userId: 0, contentVersion: 1 })
    }
}
```

**② upsertLeader — 保存总负责人并递增版本号**

同 bumpVersion，额外多了一个 `total_leader = #{totalLeader}` 字段。

**③ getReadStatus — 对比版本号判断已读/未读**

```sql
SELECT CASE
    WHEN mv IS NULL THEN false        -- 主记录不存在 → 没有公告
    WHEN uv IS NULL THEN true         -- 用户从未读过 → 有未读
    WHEN mv > uv THEN true            -- 主版本 > 已读版本 → 有未读
    ELSE false                         -- 版本相同 → 无未读
END AS hasUpdate
FROM (
    SELECT
        (SELECT content_version FROM pre_project_bulletin WHERE project_id=? AND user_id=0) AS mv,
        (SELECT content_version FROM pre_project_bulletin WHERE project_id=? AND user_id=?) AS uv
) t
```

类比前端：
```typescript
function getReadStatus(projectId: number, userId: number): boolean {
    const mainRecord  = records.find(r => r.projectId === projectId && r.userId === 0)
    const userRecord  = records.find(r => r.projectId === projectId && r.userId === userId)

    if (!mainRecord)  return false  // 没有公告
    if (!userRecord)  return true   // 从未读过 → 有更新
    return mainRecord.contentVersion > userRecord.contentVersion
}
```

**④ markRead — 对齐版本号**

```sql
INSERT INTO pre_project_bulletin (project_id, user_id, content_version, ...)
SELECT #{projectId}, #{userId}, IFNULL(t.content_version, 0), ...
FROM (SELECT content_version FROM pre_project_bulletin WHERE project_id=? AND user_id=0) t
ON DUPLICATE KEY UPDATE
    content_version = (SELECT cv FROM (...主记录版本号...) tmp)
```

类比前端：
```typescript
function markRead(projectId: number, userId: number) {
    const mainRecord = records.find(r => r.projectId === projectId && r.userId === 0)
    if (!mainRecord) return  // 主记录不存在，不用标记

    const userRecord = records.find(r => r.projectId === projectId && r.userId === userId)
    if (userRecord) {
        userRecord.contentVersion = mainRecord.contentVersion  // 对齐版本号
    } else {
        records.push({ projectId, userId, contentVersion: mainRecord.contentVersion })
    }
}
```

## 6. 实体类详解（PO / BO）

### 6.1 PO (Persistent Object) — 数据库表的类型

```java
// ==================== 后端代码 ====================

@Data                              // Lombok：自动生成 getter/setter/toString
@EqualsAndHashCode(callSuper = false)
@Accessors(chain = true)           // setter 返回 this，支持链式调用
@TableName("pre_project_bulletin") // 对应数据库表名
public class ProjectBulletin implements Serializable {

    @TableId(value = "id", type = IdType.AUTO)   // 主键，自增
    private Long id;

    private Long projectId;       // 对应表字段 project_id（自动驼峰转换）
    private Long userId;          // 对应表字段 user_id
    private String totalLeader;   // 对应表字段 total_leader
    private Integer contentVersion; // 对应表字段 content_version

    @JsonFormat(pattern = "yyyy-MM-dd HH:mm:ss")
    private Date createdAt;

    @JsonFormat(pattern = "yyyy-MM-dd HH:mm:ss")
    private Date updatedAt;
}
```

### PO vs BO 的区别

| | PO (Persistent Object) | BO (Business Object) |
|---|---|---|
| **用途** | 映射数据库表 | 接收前端请求参数 |
| **前端类比** | Prisma Model / DB Schema | 函数参数的 interface |
| **对应** | 一张表一个 PO | 一个接口一个 BO |
| **字段** | 包含所有表字段（id, 创建时间等） | 只包含前端需要传的字段 |
| **方向** | 数据库 → Java 对象 | 前端请求 → Java 对象 |

## 7. SQL 建表语句详解

```sql
CREATE TABLE IF NOT EXISTS pre_project_bulletin (
    -- ≈ TypeScript: id: number  (主键，自增)
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,

    -- ≈ TypeScript: projectId: number  (必填)
    project_id      BIGINT       NOT NULL COMMENT '项目ID',

    -- ≈ TypeScript: userId: number  (默认 0)
    -- 0 = 公告主记录；>0 = 某用户的已读记录
    user_id         BIGINT       NOT NULL DEFAULT 0 COMMENT '0=公告主记录，>0=用户已读记录',

    -- ≈ TypeScript: totalLeader: string  (最大 100 字符)
    total_leader    VARCHAR(100) DEFAULT '' COMMENT '总负责人（仅主记录有效）',

    -- ≈ TypeScript: contentVersion: number  (默认 0)
    content_version INT          NOT NULL DEFAULT 0 COMMENT '主记录=当前版本号；用户记录=已读版本号',

    -- ≈ TypeScript: updatedAt: Date
    updated_at      DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    -- ≈ TypeScript: createdAt: Date
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,

    -- 联合唯一索引：每个项目的每个用户只有一条记录
    UNIQUE KEY uk_project_user (project_id, user_id)
)
```

### 关键概念对照

| SQL 概念 | 前端类比 | 说明 |
|---------|---------|------|
| `PRIMARY KEY` | `array.findIndex()` 的快速查找键 | 唯一标识每一行 |
| `AUTO_INCREMENT` | 自增 ID | 数据库自动分配递增的 ID |
| `NOT NULL` | TypeScript 的非空类型 `string`（非 `string \| null`） | 不允许为空 |
| `DEFAULT ''` | 函数参数默认值 `totalLeader = ''` | 插入时不指定则使用默认值 |
| `COMMENT '...'` | JSDoc 注释 | 给字段加说明 |
| `UNIQUE KEY` | `Map` 的键唯一性 | `(project_id, user_id)` 组合不能重复 |
| `ON UPDATE CURRENT_TIMESTAMP` | Vue 的 `computed` 自动更新 | 记录更新时自动刷新时间 |

### "一表两用"设计

```
同一张表 pre_project_bulletin：

┌─────┬────────────┬─────────┬─────────────┬─────────────────┐
│ id  │ project_id │ user_id │ total_leader│ content_version  │
├─────┼────────────┼─────────┼─────────────┼─────────────────┤
│  1  │    100     │    0    │   "张三"    │       5          │  ← 主记录
│  2  │    100     │   1001  │     ""      │       3          │  ← 用户A已读版本=3
│  3  │    100     │   1002  │     ""      │       5          │  ← 用户B已读版本=5
└─────┴────────────┴─────────┴─────────────┴─────────────────┘

user_id = 0   → 公告本体（版本号=5，总负责人=张三）
user_id = 1001 → 用户A上次读到的版本号=3 → 5 > 3 → 有未读
user_id = 1002 → 用户B上次读到的版本号=5 → 5 = 5 → 已读
```

前端类比：
```typescript
interface BulletinRecord {
    projectId: number
    userId: number        // 0 = 主记录, >0 = 用户记录
    totalLeader: string
    contentVersion: number
}

// 判断是否已读
function hasUpdate(records: BulletinRecord[], projectId: number, userId: number): boolean {
    const main = records.find(r => r.projectId === projectId && r.userId === 0)
    const user = records.find(r => r.projectId === projectId && r.userId === userId)
    if (!main) return false
    if (!user) return true
    return main.contentVersion > user.contentVersion
}
```

### @Autowired 的本质

|------|------|

| 注册 (@Service) | 类似 `defineStore('bulletin', ...)` — 告诉框架"我存在" |

## 9. 事务 @Transactional

### 前端类比：保证一组操作"要么全成功，要么全不执行"

```java
// ==================== 后端代码 ====================

@Transactional(rollbackFor = Exception.class)
public void setUserLeave(Long userId) {
    // 操作 1：查出用户关联的项目
    List<ProjectUsers> userProjects = ...;
    // 操作 2：禁用用户
    users.setStatus(0);
    usersService.updateById(users);
    // 操作 3：删除用户的所有项目成员记录
    projectUsersMapper.deleteProjectMember(userId);
    // 操作 4：逐一 bump 公告版本号
    for (ProjectUsers pu : userProjects) {
        projectBulletinService.bumpBulletinVersion(pu.getProjectId());
    }
}
```

**如果操作 3 成功了但操作 4 报错了怎么办？**

- 没有 `@Transactional`：操作 2、3 已经生效，操作 4 失败 → 数据不一致
- 有 `@Transactional`：所有操作回滚（撤销），就像什么都没发生过一样

```typescript
// ==================== 前端等价概念（伪代码）====================

// 类比前端的状态批量更新 + 回滚
async function setUserLeave(userId: number) {
    // 保存快照，用于回滚
    const snapshot = cloneDeep(currentState)

    try {
        await disableUser(userId)
        await deleteMemberRecords(userId)
        await bumpAllBulletins(userId)
        // 全部成功 → 提交
    } catch (e) {
        // 有失败 → 回滚到快照
        Object.assign(currentState, snapshot)
        throw e
    }
}
```

## 11. 版本号已读/未读设计思路

### 核心原理

用一个**递增整数**对比来实现已读/未读判断，而不是存储每条变更记录。

```
                    content_version
公告主记录              5          ← 当前公告版本号是 5
用户A已读记录           3          ← A 上次读到版本 3 → 5 > 3 → 有未读 ✓
用户B已读记录           5          ← B 上次读到版本 5 → 5 = 5 → 已读 ✓
用户C没有记录          null        ← 从未读过 → 有未读 ✓
```

### 触发版本号递增的场景

```
项目域名变更  ──────┐
成员变更 ───────────┤     全部调用 bumpBulletinVersion()
用户离职 ───────────┤         → version++
保存总负责人 ───────┘
```

### 前端如何使用

```
用户打开项目详情页
    │
    ├─ 调用 getBulletinReadStatus(projectId)
    │   返回 { hasUpdate: true }  → 显示红点 🔴
    │
    ├─ 用户点击"群公告"按钮，打开弹窗
    │   │
    │   ├─ 并行调用 getBulletinInfo() + getProjectCoreMembers()
    │   │   获取「总负责人」和「核心成员」
    │   │
    │   └─ 弹窗展示完成后，调用 markBulletinRead(projectId)
    │       → 红点消失
    │
    └─ 用户编辑总负责人并保存
        调用 saveBulletinLeader(projectId, "新负责人")
```
